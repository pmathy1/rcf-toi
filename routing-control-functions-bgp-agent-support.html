Description

Routing Control Functions (RCF) is a language that can be used to express route filtering and attribute modification logic in a powerful and programmatic fashion.

The document covers:

  • Configurations of a RCF function for BGP points of application
  • CLI show commands to provide visibility into the operational status of RCF
  • The protocol attributes supported for BGP points of application

Additional documentation covering aspects of the RCF language can be found in the Resources section at the end of this document.

Configuration

Applying RCF to BGP points of application

RCF functions can be used for filtering routes and updating route attributes at various points of application in BGP.

Instance configuration

The following configurations allow filtering and modifying path attributes while instantiating IPv4 Unicast and IPv6 Unicast AFI/SAFI paths from the respective sources (the RIB, VPN AFI/SAFI, etc).

The supported path attributes and built-in functions for unicast paths are listed here.

Redistribute statement

connected

Supported since EOS-4.24.2F
Syntax 
router bgp <ASN>
   [ vrf <NAME> ]
      redistribute connected rcf <FUNCTION_NAME>()
Example usage 
router bgp 64500
   redistribute connected rcf CONNECTED_POLICY()

dynamic

Supported since EOS-4.29.1F
Syntax 
router bgp <ASN>
   [ vrf <NAME> ]
      redistribute dynamic rcf <FUNCTION_NAME>()
Example usage 
router bgp 64500
   redistribute dynamic rcf DYNAMIC_POLICY()

isis

Supported since EOS-4.26.1F
Syntax 
router bgp <ASN>
   [ vrf <NAME> ]
      redistribute isis ( level-1 | level-2 | level-1-2 ) rcf <FUNCTION_NAME>()
Example usage 
router bgp 64500
   redistribute isis level-1 rcf LEVEL_1_POLICY()

static

Supported since EOS-4.24.2F
Syntax 
router bgp <ASN>
   [ vrf <NAME> ]
      redistribute static rcf <FUNCTION_NAME>()
Example usage 
router bgp 64500
   redistribute static rcf STATIC_POLICY()

MPLS L3 VPN import

The IMPORT_FUNCTION_NAME() function applies to the unicast route created on import. The unicast route attributes will be populated with the BGP Customer Attribute Set (RFC 6368) if it exists, otherwise the VPN path attributes will be used.

The VPN_FUNCTION_NAME() function applies to the VPN route that is imported using the provider attributes. Modifications to path attributes in the VPN_FUNCTION_NAME() function don't apply to the provider path. See the BGP MPLS L3 VPN TOI for more information.

Supported since EOS-4.27.2F
Syntax 
router bgp <ASN>
   vrf <NAME>
      route-target import ( vpn-ipv4 | vpn-ipv6 ) rcf <IMPORT_FUNCTION_NAME>() [ vpn-route filter-rcf <VPN_FUNCTION_NAME>() ]
Example usage 
router bgp 64500
   vrf VRF1
      route-target import vpn-ipv4 rcf IMPORT_POLICY() vpn-route filter-rcf FILTER_VPN_ROUTES()

L3 EVPN import

The IMPORT_FUNCTION_NAME() function applies to the unicast route created on import. See the BGP L3 EVPN TOI for more information.

Supported since EOS-4.27.2F
Syntax 
router bgp <ASN>
   vrf <NAME>
      route-target import evpn rcf <IMPORT_FUNCTION_NAME>()
Example usage 
router bgp 64500
   vrf VRF1
      route-target import evpn rcf IMPORT_POLICY()

MPLS L3 VPN export

The EXPORT_FUNCTION_NAME() function applies to the VPN route created on export. The unicast route path attributes are pushed into a BGP Customer Attribute Set if the iBGP as PE-CE Protocol for BGP/MPLS L3 VPNs (RFC 6368) is configured, these attributes are not modified in policy. The VRF_FUNCTION_NAME() function applies to the unicast route that is to be exported. Modifications to path attributes in the VRF_FUNCTION_NAME() function don't apply to the provider path. See the BGP MPLS L3 VPN TOI for more information.

Supported since EOS-4.28.2F
Syntax 
router bgp <ASN>
   vrf <NAME>
      route-target export ( vpn-ipv4 | vpn-ipv6 ) rcf <EXPORT_FUNCTION_NAME>() [ vrf-route filter-rcf <VRF_FUNCTION_NAME>() ]
Example usage 
router bgp 64500
   vrf VRF1
      route-target export vpn-ipv4 rcf EXPORT_POLICY() vrf-route filter-rcf FILTER_VRF_ROUTES()
      route-target export vpn-ipv6 rcf EXPORT_POLICY() vrf-route filter-rcf FILTER_VRF_ROUTES()

LDP tunnel redistribution

LDP tunnel redistribution is only available for the IPv4 Labeled Unicast address family.

Supported since EOS-4.28.0F
Syntax 
address-family ipv4 labeled-unicast
   tunnel source-protocol ldp rcf <FUNCTION_NAME>()
Example usage 
router bgp 64500
   address-family ipv4 labeled-unicast
      tunnel source-protocol ldp rcf LDP_POLICY()

Dynamic prefix list configuration

The DPL_FUNCTION_NAME() function is evaluated on BGP routes in a given VRF in order to activate the dynamic prefix list. The supported path attributes and built-in functions for dynamic prefix list configuration are listed here.

Supported since EOS-4.30.2F
Syntax 
dynamic prefix-list <DYN_PFX_LIST>
   match rcf <DPL_FUNCTION_NAME>()
Example usage 
dynamic prefix-list DYN1
   match rcf TRIGGERING_ROUTE_EXISTS()

Show commands

To view the status of RCF functions in BGP, use the show bgp rcf [<FUNCTION_NAME>] command.

Arista#show bgp rcf
Function                       Status                         Detail
------------------------------ ------------------------------ ------------------------------
ACCEPT_ALL()                   active
DENY_ALL()                     active
USES_MISSING_PFXL()            prefix_list_v4 not found       MISSING_PFXL


Arista#show bgp rcf ACCEPT_ALL()
Function                       Status                         Detail
------------------------------ ------------------------------ ------------------------------
ACCEPT_ALL()                   active
Arista#

The possible status values are enumerated below.

  • active: the RCF function is fully configured and is available for use
  • reference not found: the RCF function refers to an external construct (e.g. as path list, community list, or prefix list) which is not defined. All routes evaluated against such an RCF function are not accepted. The undefined reference name can be found in the “Detail” column.
  • missing symbol: the RCF function is in the process of getting programmed. This should only appear transiently and is not persistent.

Starting from EOS-4.26.0F, the show bgp neighbors command will display the RCF functions configured on inbound and outbound for BGP neighbors. If both a route map and an RCF function are configured for the same point of application (inbound or outbound), the show command will indicate that the RCF configuration takes precedence.

Arista#show bgp neighbors
BGP neighbor is 1.0.1.2, remote AS 500, internal link
…
  Inbound route map for IPv4 unicast is RM1 (disabled due to overriding RCF function)
  Inbound RCF function for IPv4 unicast is ACCEPT_ALL()
  Outbound RCF function for IPv4 unicast is ACCEPT_ALL()
…

Debugging

With the EOS-4.27.2F release, the RCF evaluation engine provides a textual representation explaining the control flow of a given evaluation of an RCF function.

 

The CLI commands to debug the RCF function applied at points of application are defined below. It is also possible to specify an RCF function that is not currently applied at the point of application to determine its behavior if it were applied.

Documents that cover the output of the CLI (“Routing control functions - Language and configuration”) and the usage of the CLI (“Policy Debugging CLI”) can be found in the “Resources” section.

Limitations

  • Only one of either a route map or an RCF function may be applied to a BGP point of application. If both are configured, only the RCF function takes effect.
  • bgp missing-policy direction … action … configuration is ignored and is not supported for RCF functions.
  • as_path prepend operation does not take effect for IBGP peers.
  • RCF is only supported in the multi-agent routing protocol model. There is no support for RCF in the ribd routing protocol model.
  • BUG680521 - There is no PROTO value to use with source_protocol attribute to identify LDP and IS-IS SR redistributed paths into  BGP Labeled Unicast.
